## 🎯 目的

本规范用于约束 `Philosophy` 尤其是 `Philosophy/model` 下各类模型文档的结构与写法，保证：

- 各层级、各视角之间的结构一致；
- 新增内容可以自然挂接到现有的思维导图、矩阵、证明树、决策树与公理定理索引中；
- 方便未来大规模重构与自动化生成导航。

## 一、文件命名与编号

### 1.1 命名规则

```text
两位序号-中文标题.md
01-主题层级模型.md
02-思维导图总览.md
...
10-DKB公理与定理索引.md
11-写作与结构规范.md
```

- **视角文档（view01-06）**：保持现有命名，不再新增编号。
- **模型文档（model/0X-..）**：按功能分组，避免插队改名（优先在说明中扩展）。

### 1.2 逻辑编号对应

- **01**：七层结构总纲（哲学→技术→评估→实施→竞争→证明→形式化数学基础）
- **02–03**：图与矩阵（结构 & 比较层）
- **04–05**：树（证明树 & 决策树）
- **06**：时间线（历史/趋势）
- **07–08**：术语与案例（支撑层）
- **09**：跨模块映射（Philosophy ↔ docs/concepts）
- **10**：正式公理/定理索引（形式化层）
- **11**：当前文件（元规范层）

## 二、标题与层级结构

### 2.1 标题层级

- 顶层标题：`#`（文件名）
- 一级章节：`##`（配图标，如 `## 🏗️ 第一层：...`）
- 二级小节：`###`（使用数字编号，如 `### 1.1 ...`）
- 三级以下：尽量用列表代替，除非有必要。

### 2.2 一致性要求

- **层级模型（01）**：必须以"第一层、第二层…"形式出现，且顺序固定：
  1. 哲学基础层
  2. 技术架构层
  3. 评估指标层
  4. 实施路径层
  5. 竞争格局层
  6. 形式化证明层
  7. 形式化数学基础层（范畴论/类型论/信息论/系统论映射）
- **其他模型文档**：若引用这七层，必须使用相同命名与顺序，以便交叉跳转。

## 三、内容分布约定（放在哪个文件）

### 3.1 视角 vs 模型

- **view01-06**：叙事 + 论证（文字为主），可以包含 mermaid 图，但不承载“唯一版本”的结构。
- **model/**：承载“结构化真相”：
  - 01：结构骨架（七层，含形式化数学基础层）
  - 02：所有导图的"收纳盒"（含思维导图7：信息-表征-意义-转换统一视角）
  - 03：所有矩阵的"收纳盒"（含矩阵11：范畴论/类型论等哲科模型对比、矩阵12：五阶段成熟度对比）
  - 04：所有证明树的"收纳盒"（含证明树10：不变量与演化边界）
  - 05：所有决策树的“收纳盒”
  - 06：统一时间线
  - 07：统一术语表
  - 08：统一案例表
  - 09：跨模块映射表
  - 10：A/L/T 索引（形式化）
  - 11：写作与结构规范
  - 12：风险与反证总览
  - 13：实施指南详细版

### 3.2 新内容应该放哪里？

- **新的图/导图** → `02-思维导图总览.md`，并在相关 view 中引用，而不是只写在 view 里。
- **新的对比/分类维度** → `03-概念多维对比矩阵.md` 新增一张矩阵（当前已有12个矩阵，矩阵9为"风险与反证"，矩阵12为"五阶段成熟度对比"）。
- **新的证明/论证结构** → `04-证明树图总览.md` 新增证明树，并在 view 中以文字复述。
- **新的决策流程/分叉** → `05-决策树图总览.md` 新增决策树。
- **新的长周期演化分析** → `06-时间线演进模型.md` 中补充新节点与阶段划分。
- **新的术语** → 在 `07-术语表与概念索引.md` 中添加条目，并回链到首次使用的 view/model。
- **新的案例** → 在 `08-案例研究索引.md` 中添加案例卡片。
- **新的形式化命题** → 在 `10-DKB公理与定理索引.md` 中添加 A/L/T 条目，并在相关文档中引用该编号。

## 四、形式化与 A/L/T 编号使用规范

### 4.1 命名约定

- **Axioms（公理）**：`A1, A2, ...`
- **Lemmas（引理）**：`L1, L2, ...`
- **Theorems（定理/推论）**：`T1, T2, ...`

### 4.2 使用方式

- 在 view/model 文本中首次给出自然语言表述 → 在 `10-DKB公理与定理索引.md` 中补上编号与正式摘要。
- 在其他文档引用时，使用：
  - “见公理 A1（语义鸿沟公理）”
  - “由 T1（DKB 基础设施定理）可得…”
  并在需要时加上链接或行内说明。

### 4.3 证明位置约定

- 核心证明应在 `model/04-证明树图总览.md` 中有对应证明树；view 只保留人类可读版本。
- 定理的“正式陈述 + 引用列表”统一收口在 `10-DKB公理与定理索引.md`。

## 五、Markdown 与代码块规范

### 5.1 标题与空行

- 标题上下各留一行空行（除文件顶部外）。
- 不在标题行末尾留空格。

### 5.2 代码块

- 所有 fenced code 必须带语言标识：

```text
    ```text
    ... 内容 ...
    ```

```

- Mermaid 图使用：

```text
    ```mermaid
    graph TD
      ...
    ```

```

### 5.3 列表缩进

- 同一层级的列表项左侧不额外缩进：

```text
- 项目1
- 项目2
  - 子项目2.1
```

## 六、与其他目录的协同

### 6.1 与 `Philosophy/00-主题总览与导航.md`

- 00 文件是“视角总览 + 进度矩阵”，**不直接承载结构真相**，而是指向 `model/`。
- 任何结构变化（新增矩阵/证明树/决策树），应同步更新：
  - 00 文件中的“结构一致性与展开进度”表；
  - `model/README.md` 的索引。

### 6.2 与 `docs/` / `concepts/` 的协同

- 所有从教学/理论角度引用 Ontology/DEX/Palantir 模型的章节，应优先：
  - 通过 `09-跨模块映射索引.md` 找到对应的 view/model；
  - 使用 A/L/T 编号和矩阵/树编号进行引用（而非重新发明命名）。

## 七、以后扩展时的 Checklist

在新增/修改任何 `Philosophy/model` 文档前，建议自查：

1. **是否已有对应类型的集中收纳文件？**
   - 若是图 → 02；矩阵 → 03；证明 → 04；决策 → 05；指标/历史 → 01/06 等。
2. **是否需要在 00 总览/README 中同步？**
3. **是否需要在 07/08/09/10 中补充术语/案例/映射/公理定理？**
4. **是否遵守了标题/代码块/列表的 Markdown 规范？**

做到这 4 点，可以基本保证新内容能够自然嵌入现有体系，而不会破坏整体结构。
